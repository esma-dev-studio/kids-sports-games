# サブエージェント体制（このゲームを作るための布陣）

作業を **設計監修 → 実装 → 品質 → 出荷** に分け、`Workflow`（Claude Code の多エージェント・オーケストレーション）で **パイプライン**（案件ごとに設計→実装→QAが流れる）＋要所で **並列** を回す。ディレクター（メインのClaude）が全体を統括し、各専門エージェントへ委譲する。

## 布陣（3レイヤー・8ロール）

| レイヤー | エージェント | 責務 | 主な使い方 |
|---|---|---|---|
| 統括 | **ゲームディレクター** | 設計ロック・優先度判断・各レビューの統合・ユーザーとの合意形成 | メインループ |
| 設計 | **スポーツ科学／教育監修** | 能力転移主張の検証・年齢適合・光過敏/モーション配慮・誇大コピーの排除 | 各機能を並列レビュー |
| 実装 | **ゲームコア実装** | Canvasゲームループ・状態機械・GO/NO-GO判定・スコア/コンボ・DDA（適応難度） | パイプライン主軸 |
| 実装 | **UI/UXシェル実装** | React外殻・HUD・メニュー・リザルト・レスポンシブ/タッチ・アクセシビリティ | コアと並行 |
| 実装 | **アート・演出・サウンド** | 演出の気持ちよさ・パーティクル・SE・キャラ・二重符号化（色＋形）・カラーパレット | コアと並行 |
| 品質 | **QA／プレイテスト** | preview系ツールで実機検証・性能（60fps）・端末横断・理不尽/バグ検出 | 各実装の直後に検証 |
| 品質 | **コードレビュー** | 差分の正確性・簡素化・再利用（`/code-review`） | マージ前 |
| 出荷 | **ビルド・デプロイ** | Vite設定・GitHub Pages公開・パス基準の調整 | 最終段 |

## オーケストレーションの型

```
Phase 0  設計ロック      ディレクター＋監修で GDD を確定（未決の論点を潰す）
Phase 1  技術基盤        ビルド担当が Vite+React+TS+Canvas 雛形とデプロイ経路を用意
Phase 2  コア実装        コア担当が状態機械・GO/NO-GO・スコア（＝MVPの心臓部）
Phase 3  差別化の核      見きわめボーナス／がまん連鎖コンボ（ここが命）
Phase 4  演出・UX        アート＋UIが並行で juice とシェルを載せる
Phase 5  監修・QA        監修＋QAが並列レビュー → 指摘をコアへ差し戻し
Phase 6  出荷           レビュー通過後にデプロイ
```

- **並列にすべき所**：Phase 4（アート／UI）、Phase 5（監修／QA／コードレビューは独立視点なので同時に）。
- **パイプラインにすべき所**：機能単位（1機能が 実装→QA→レビュー を独立に流れる。全機能の足並みを揃える必要はない）。
- **アダーサリアル検証**：転移主張とアクセシビリティは「反証を試みる」姿勢の監修エージェントで二重チェックし、誇大表現を落とす。

## 実ファイル化について

この体制は `.claude/agents/*.md`（各エージェント定義）として実際に構築でき、以後 `Agent` ツールから名前で呼び出せる。構築する場合は各定義に「役割・持ちツール・出力フォーマット・禁止事項（例：転移を誇張しない）」を明記する。

> 現状は設計提示まで。実ファイル化と開発着手は、ゲームの最終決定後に実施予定。
